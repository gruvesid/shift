Flow Extraction from Salesforce

backend/app/code_converter_router.py

Salesforce flows are extracted via the Tooling API (not the REST API) because the Metadata field on the Flow object is only accessible through Tooling SOQL.

Flow ID Resolution — 4-Strategy Fallback

Function: _resolve_flow_version_id() (~line 120)

Why needed The Salesforce component list returns DefinitionId (starting with 300), but the Tooling API Metadata field requires the active version ID (starting with 301). The resolution chain handles both.
PriorityStrategyAPI Call
PRIMARYREST FlowDefinitionView by ApiNameSELECT ActiveVersionId, LatestVersionId FROM FlowDefinitionView WHERE ApiName = '{name}'
2Direct Tooling GET (ID starts with 301)Tooling GET /Flow/{id}
3REST sobjects FlowDefinitionView/services/data/v59.0/sobjects/FlowDefinitionView/{id}
4Tooling SOQL by DefinitionIdSELECT Id FROM Flow WHERE DefinitionId = '{id}' ORDER BY VersionNumber DESC LIMIT 1

Metadata Fetch

Function: _fetch_flow_metadata() (~line 202)

# Primary query — Metadata field only accessible via Tooling SOQL
SELECT Metadata FROM Flow WHERE Id = '{version_id}'

# Secondary query — internal fields
SELECT Id, FullName, Status, ProcessType FROM Flow WHERE Id = '{version_id}'

Returns a dict with full Metadata + internal fields prefixed with _ (_version_id, _process_type, etc.).

Source Format Returned

Function: _fetch_flow_source() (~line 334)
Returns __FLOW_META__:{json} where json contains:

  • metadata — full Flow.Metadata dict (variables, decisions, records, screens, etc.)
  • text — formatted text for LLM (sections labelled VARIABLES, DECISIONS, RECORDS, etc.)
  • raw_json — pretty-printed JSON for the XML view tab

Flow Rulebook

backend/app/models/rulebook.py backend/app/power_automate_router.py

DB Model — rulebooks table

ColumnTypeValue for Flow
component_typeVARCHAR(50) UNIQUE"flow"
titleVARCHAR(200)"Salesforce Flow → Power Automate Cloud Flow (JSON)"
system_promptTEXTExpert migration engineer prompt
rulesTEXT6-section conversion rules
updated_atDATETIMEAuto-set

Rules — 6 Sections

SectionContent
1 — Flow Type Detectionprocess_type + trigger_type → flow_type table (Screen Flow, Automated, Scheduled, etc.)
2 — Trigger InputsfieldType + dataType → trigger_inputs schema (InputField+String→string, DropdownBox→string, DisplayText→SKIP)
3 — Action MappingRecordCreate→AddRow, RecordUpdate→UpdateRow, RecordLookup→GetRow/ListRows, Decision→Condition, Loop→Foreach, emailSimple→SendEmail
4 — Field Name RulesAlways use Dataverse_Column from field mapping; entity logical names from Dynamics_Object; NEVER Salesforce __c names
5 — Formula MappingTODAY()→utcNow(), IF(c,t,f)→if(), TEXT()→string(), a=b→equals(), etc.
6 — Output JSON StructureExact JSON schema the LLM must return (inside <converted>...</converted>)

Critical Rules

⚠ Must follow
  • AddRow row values MUST use simple @{triggerBody()?['field']} references only — no if(), no equals() nested inside row values
  • Always include "new_dynamiccheckbox": false in every Account AddRow action
  • Choice/OptionSet fields → integer DY_picklist codes, never string labels

Deploy to Power Automate (Dataverse API)

backend/app/power_automate_router.py
Why Dataverse API and NOT PA Management API The PA Management API (api.flow.microsoft.com) does not support Service Principal (app-only) tokens — it requires delegated user auth. Calling it with an SP token returns ClientScopeAuthorizationFailed. The Dataverse Workflow API (category=5) works with SP auth using scope {env_url}/.default.

Full Deploy Pipeline

POST /power-automate/deploy ↓ Create DeploymentLog (status="running") ← FIRST — so all errors appear in history ↓ _parse_flow_json(converted_code) ├── Strip <converted>...</converted> tags ├── Try direct JSON.parse ├── Regex search for JSON blocks (```json, {"flow_name":, etc.) └── Fallback: bracket-depth scan ↓ _is_structured_json(flow_json)? ├── YES → _build_clientdata_from_structured() └── NO → _build_clientdata_from_logicapps() → _normalise_definition() ↓ _deploy_via_dataverse_api() ↓ Update DeploymentLog (status, flow_url, log_text) Update ConvertedItem (status="deployed"|"manual")

Dataverse Workflow API — Call Sequence

scope = f"{env_url}/.default"
token = msal.ConfidentialClientApplication(...).acquire_token_for_client(scopes=[scope])

# 2. Check for existing flow (DELETE + re-CREATE — PATCH clientdata is unreliable)

Body: {
  "name": flow_name,
  "category": 5,           # Modern Flow (Power Automate)
  "type": 1,               # Definition
  "primaryentity": "none",
  "clientdata": "<json-string>",  # double-serialized!
  "statecode": 0,          # Draft
  "statuscode": 1
}

# 4. Extract workflow GUID from OData-EntityId response header

# 5. Auto-activate
PATCH {wf_api}({wf_id}) → {statecode: 1, statuscode: 2}

Fix Button — LLM Auto-Fix on Error

backend/app/d365_deploy_router.py frontend/src/components/CodeConverterPage.js
What it does After a deploy failure, the 🔧 Fix button sends the full error context to the LLM and auto-updates the converted code so the user can deploy again immediately — no manual editing required.

Backend Endpoint

Endpoint: POST /d365-deploy/saved/{item_id}/fix
Function: fix_saved()
Request: { "error_context": "<deploy error message>" }

System Prompt

You are an expert Salesforce → Microsoft Dynamics 365 migration engineer.
A converted component failed to deploy. Analyse the error and produce a corrected version.

Rules:
1. Fix ONLY the issues described in the error. Do not refactor unrelated code.
2. For C# plugins: keep IPlugin structure, recursion guard, Contains() guards, using statements.
3. For Flow JSON: keep same JSON structure, fix field names / action types / expressions.
4. For PCF TypeScript: keep StandardControl<IInputs,IOutputs> structure.
5. Return COMPLETE fixed code — not a diff.

Respond ONLY in:
<fixed_code>...</fixed_code>
<explanation>...</explanation>

User Prompt Includes

  • COMPONENT TYPE and COMPONENT NAME
  • ORIGINAL SOURCE CODE (Salesforce Apex/Flow) — from ConvertedItem.sf_source
  • CURRENT CONVERTED CODE (the version that failed)
  • DEPLOY ERROR (exact error from Dataverse or PA API)

LLM: Uses get_llm_for_task(db, "convert") — highest-quality model configured for conversion.
Auto-update: If fixed_code differs, ConvertedItem.d365_output is updated and status reset to "converted". LLM usage logged to llm_usage table.

Frontend Behavior

Function: handleFix() — _getLastError() collects error from:

  • deployResult.errors[] — D365 plugin/PCF deploy errors
  • deployResult.manual_instructions — manual deploy message
  • paResult.error — Power Automate deploy error
  • paResult.log_text — full deploy log

After fix: convertedCode updated, deployResult and paResult reset, green banner shows explanation + model + token cost.

Visual cue The Fix button turns red with warning styling when there is an active unresolved deploy error.

Key Architecture Decisions

Decision 1 — Dataverse API over PA Management API

The PA Management API requires delegated user auth and rejects SP tokens with ClientScopeAuthorizationFailed. The Dataverse Workflow API (/api/data/v9.2/workflows) accepts SP auth via {env_url}/.default — matching exactly how the reference POC AI project works.

Decision 2 — DELETE + re-CREATE (not PATCH)

PATCH on an existing workflow's clientdata field returns TemplateValidationError. The reliable pattern is: check for existing flow by name → DELETE → POST new. Guarantees a clean state every time.

Decision 3 — Log Created Before Deployment

The DeploymentLog row is committed to DB before any attempt. This ensures all failures — credential errors, JSON parse failures, Dataverse 400/401 — are always visible in the History modal.

Decision 4 — Two-Format Clientdata Pipeline

Two paths for backward compatibility: Structured JSON (POC AI format) → _build_clientdata_from_structured(); Legacy Logic Apps (old ApiConnection/PostItem) → _normalise_definition() → wrap.

Decision 5 — Fix Before Redeploy (not Retry)

Rather than silently retrying, the Fix button sends the actual error to the LLM for targeted correction. The fixed code replaces the stored conversion and the user explicitly re-deploys — full visibility and control.
